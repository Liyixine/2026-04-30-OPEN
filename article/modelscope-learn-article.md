# 【Intel AI PC创新应用征文】基于 OpenVINO 的端侧现场交付状态自动说明助手

## 一、项目基础信息

- **项目名称**：端侧现场交付状态自动说明助手（Delivery Proof Assistant）
- **应用方向**：视觉理解（VLM）+ 语音合成（TTS）+ 语音识别（ASR）+ Agent Skill
- **参考 Baseline**：OpenVINO ModelScope Workshop（Lab 1 / Lab 2 / Lab 3）
- **参考仓库**：https://github.com/openvino-dev-samples/modelscope-workshop
- **目标设备**：Intel AI PC / 支持 OpenVINO 的本地设备
- **灵感流 Notebook**：`delivery-proof-assistant.ipynb`
- **项目仓库**：https://github.com/Liyixine/delivery-proof-assistant

## 二、应用场景说明

### 一个真实的日常痛点

快递放在前台了，发个消息告诉同事——这件事听起来简单，实际操作却经常出问题。

配送员拍了一张照片发给收件人，但只说"放前台了"。收件人走到前台，面对一排柜子和几个快递堆在一起，还是分不清哪个是自己的、到底放在哪个位置。接下来就是反复追问："你说的前台是哪个桌子？""旁边有什么标志？"

这种场景不只发生在办公室。在医院护士站，家属送来的药品袋需要明确交接到哪个位置；在工厂仓库，物料箱送到工位后，班组需要确认放在了标线内还是货架旁。

这些场景有几个共同的特点：

1. **网络条件受限**——工厂车间、医院内网、政企园区往往没有稳定的外网连接
2. **隐私要求高**——照片里可能出现门牌号、科室名、工位编号，甚至患者信息
3. **人工描述成本高**——现场人员每次都要手打一段位置说明，效率很低
4. **沟通误差频繁**——文字描述不够精确，收件人经常需要二次追问

### 我们要解决什么

把"拍照 + 说明放在哪里"这件事自动化。配送员只需要拍一张照片，系统就能自动识别照片里的物品、放置位置和周围参照物，然后生成一条可以直接发给收件人的通知，同时输出一段适合语音播报的短文本。

关键是：**所有推理都在本地完成**，照片不需要上传到任何外部服务。

```text
交付现场照片
  -> 本地 VLM 理解物品、位置、参照物
  -> 自动生成送达通知
  -> 输出 TTS 播报文本
  -> 支持二次追问
  -> 生成 Markdown 留档
```

## 四、整体方案设计

系统的数据流比较直观：一张照片进来，经过四层处理，最终产出多种格式的交付凭证。

```text
[交付现场图片 + 业务上下文]
         ↓
[OpenVINO VLM 图片理解]  ← Qwen3-VL-4B, INT4 量化
         ↓
[交付信息结构化 + 场景模板匹配]
         ↓
[收件人通知 / TTS 播报音频 / ASR 回读验证 / Markdown 留档]
```

### 4.1 三个典型场景

第一版覆盖了三个最常见的交付场景，每个场景在通知语气和关注重点上有所不同：

- **企业园区**——文件袋、快递箱、办公设备。通知语气偏礼貌简洁，重点描述前台、货架、会议室等容易找到的参照物。
- **医院科室**——药品袋、检验材料、家属物品。语气正式克制，只描述交接位置和物品状态，避免涉及任何医疗建议。
- **工厂仓库**——物料箱、工具箱、备件。语气偏工单风格，优先描述工位编号、货架位置、地面标线区域。

选择这三个场景而非外卖配送，是因为它们更能体现端侧部署的价值：不依赖外部平台 API，在内网环境下独立运行，隐私保护的理由也更充分。

### 4.2 输出结构

不论哪个场景，系统最终输出统一的 JSON 结构：

```json
{
  "scene": "enterprise_office",
  "item": "文件袋 1 个",
  "location": "A 座 3 楼前台右侧桌面",
  "landmark": "蓝色文件架、访客登记牌",
  "status": "包装完整，已放置妥当",
  "recipient_message": "张工您好，您的文件袋已送达 A 座 3 楼前台右侧桌面，旁边可参考：蓝色文件架、访客登记牌。请您方便时领取。",
  "tts_text": "张工您好，您的文件袋已放在A座三楼前台右侧桌面。旁边有蓝色文件架、访客登记牌，请您方便时领取。",
  "followup_answer": "可以重点查看 A 座 3 楼前台右侧桌面，附近参照物是蓝色文件架和访客登记牌。"
}
```

## 五、Baseline 跑通过程与踩坑记录

本项目参考官方 OpenVINO ModelScope Workshop 的三个 Lab：

- **Lab 1**：Qwen3-VL 多模态视觉语言理解
- **Lab 2**：Qwen3-ASR 语音识别
- **Lab 3**：Qwen3-TTS 语音合成

官方提供的模型已经预先导出为 OpenVINO IR 格式，可以直接从 ModelScope 下载并加载推理，省去了模型转换的步骤。

### 踩坑 1：`KeyError: 'qwen3_vl'`

第一次加载 VLM 模型时遇到了这个错误。排查后发现原因是环境中的 `transformers` 和 `optimum-intel` 版本太旧，不认识 Qwen3-VL 这个模型架构。

解决方法是严格对齐官方 workshop 的依赖组合：

- `openvino==2026.0`
- 官方固定 commit 的 `optimum-intel`（通过 git 安装）
- `transformers>=4.57.0`
- `torch==2.8`

重新安装依赖并重启 Kernel 后问题解决。这里的教训是：**对于新模型架构的支持，依赖版本必须严格匹配，不能只升级单个包**。

### 踩坑 2：TTS 和 ASR 的 transformers 版本冲突

Qwen3-TTS 官方源码依赖 `transformers==4.57.3`，而 Qwen3-ASR 依赖 `transformers==4.57.6`。如果在同一个 Python 进程中先加载 TTS 再加载 ASR，ASR 的 processor 初始化会失败：

```text
RuntimeError: Processor not loaded. Cannot transcribe.
```

我的解决方案是把 ASR 放到独立的 Python 子进程中运行。项目中封装了一个 `transcribe_audio_subprocess` 函数，它会启动一个新的 Python 进程来执行 ASR 推理，结果通过 JSON 文件传回。这样主进程中的 VLM 和 TTS 不受影响，ASR 也能正常工作。

VLM 的核心调用方式参考官方 Lab 1：

```python
from optimum.intel.openvino import OVModelForVisualCausalLM
from transformers import AutoProcessor
```

## 六、从 Baseline 到交付助手的创新改造

官方 Baseline 的定位是验证单个模型的基本能力——VLM 能看懂图片，ASR 能听懂语音，TTS 能合成语音。它们各自独立，没有串成一个完整的应用。

本项目在此基础上做了以下改造：

### 6.1 场景收束：从通用问答到交付凭证

Baseline 中的 VLM 演示是开放式图片问答，什么都能问。但在实际应用中，"什么都能问"往往意味着输出不可控。我把 VLM 的能力收束到一个具体任务上——**识别交付现场的物品、位置和参照物**，并通过精心设计的 prompt 来约束输出格式。

### 6.2 结构化输出：从自由文本到字段拆解

Baseline 输出的是一段自然语言描述。但交付场景需要把信息拆解成独立字段（物品、位置、参照物、状态），才能分别用于生成通知、TTS 播报和留档记录。

### 6.3 多场景模板：不同行业，不同语气

企业前台的通知应该礼貌简洁，医院交接应该正式克制，工厂工位应该像工单记录。我为三个场景分别设计了通知模板，让生成的文本符合各自行业的沟通习惯。

### 6.4 TTS 播报文本独立生成

完整通知适合作为文字消息发送，但直接拿来做语音播报会显得像在朗读清单。因此 TTS 播报文本是单独生成的，只保留最关键的信息——收件人、物品、位置和简短的领取提示。

### 6.5 多模态链路串联

把 VLM、TTS、ASR 三个独立能力串成了一条完整的处理链路：图片理解 → 通知生成 → 语音播报 → 回读验证。其中 ASR 既可以验证 TTS 的播报质量，也预留了作为"语音追问"入口的扩展空间。

### 6.6 Skill 封装

将整个能力封装为 OpenClaw Skill，使 Copaw 中的 Agent 可以把本项目当作一个外部工具来调用：克隆代码、安装依赖、下载模型、执行推理，全程自动化。

## 七、运行效果展示

### 7.1 输入素材

Notebook 内置了三张示例图片，分别对应三类交付场景：

- `samples/office_001.png`：企业园区——文件袋放在 A 座 3F 服务台前台桌面
- `samples/hospital_001.png`：医院科室——医疗物品袋放在 9F 护士站物品交接区
- `samples/warehouse_001.png`：工厂仓库——三箱物料放在 B 区 04 工位旁的黄色标线区域

样例清单 `samples/manifest.example.json` 只提供必要的业务上下文（场景、收件人、物品提示、补充备注），**不预置最终回复**。所有通知、TTS 文本和追问回答都由 Notebook 运行时实时生成。

### 7.2 VLM 输出格式优化

在第一轮真实推理中，我发现 VLM 有时会把字段值输出成 Python 列表的形式：

```json
{
  "location": "['前台接待台', 'A座3F服务台前']",
  "landmark": "['A座3F服务台', '大理石前台', '走廊入口']"
}
```

这会导致生成的通知读起来像在念代码：

```text
张工您好，文件袋 1 个已送达['前台接待台', 'A座3F服务台前']...
```

问题的根源不在模型的识别能力，而在 prompt 对输出格式的约束不够明确。我做了两层修复：

1. **Prompt 层面**：在 VLM prompt 中明确要求"所有字段值必须是自然语言字符串，不要输出数组、列表或 Python 表达式"。
2. **后处理层面**：在代码中兼容数组、列表字符串和空数组，统一清洗为自然语言表达。

优化后的通知读起来流畅多了：

```text
张工您好，文件袋 1 个已放在 A座3F服务台前台桌面。
旁边有蓝色文件架、访客登记牌和绿植，方便您查找。
请您方便时领取。
```

### 7.3 TTS 与 ASR 运行效果

**TTS 语音合成**已接入官方 Lab 3，使用 `Qwen3-TTS-CustomVoice-0.6B-fp16-ov` 模型。Notebook 中会展示生成的音频控件、音频时长、推理耗时和实时率（RTF）。TTS 播报文本经过独立优化，语气更接近现场人员的自然提醒，而不是机械地朗读留档信息。

**ASR 语音回读**已接入官方 Lab 2，使用 `Qwen3-ASR-0.6B-fp16-ov` 模型。由于与 TTS 存在 transformers 版本冲突，ASR 在独立子进程中运行。它的主要用途是回读验证 TTS 音频的播报质量，同时也预留了作为语音追问入口的扩展能力。

## 八、Skill 封装展示

项目已将完整能力封装为 OpenClaw Skill，文件位于：

```text
skills/delivery-proof-assistant/SKILL.md
```

与常见的"把 prompt 喂给小模型"不同，这个 Skill 的设计理念是**让 Copaw 中的 Agent 把整个项目当作一个外部工具来调用**。Skill 文件中包含了完整的操作指引：

1. **环境准备**：克隆代码仓库、安装依赖、下载 VLM 模型
2. **核心功能调用**：传入照片和业务上下文，获取结构化交付结果
3. **扩展能力**：TTS 语音合成、ASR 回读验证、批量处理、Gradio 交互界面
4. **故障排除**：针对常见依赖冲突问题的解决方案

Agent 可以根据用户的实际需求，灵活选择调用哪些功能——可以只做图片分析和通知生成，也可以走完整的"图片 → 通知 → 语音 → 回读"全链路。

## 九、总结与展望

### 做了什么

本项目从一个很小但很真实的需求出发——"拍照后自动说明东西放在哪里"，把官方 VLM、TTS、ASR 三个 Baseline 的能力重新组合，构建了一个面向弱网和隐私敏感场景的端侧交付凭证生成助手。

相比单纯复现 Baseline，这个项目更侧重于：

- **聚焦具体行业场景**，而非通用能力展示
- **强调弱网和内网可用性**，所有推理在本地完成
- **注重隐私保护**，照片和数据不离开本地设备
- **串联多模态能力**，形成完整的应用链路
- **封装为可复用的 Skill**，降低其他开发者的接入成本

### 端侧部署的一些体会

OpenVINO 的量化模型在普通 CPU 上的推理速度已经完全可以满足这类非实时场景的需求。INT4 量化后的 Qwen3-VL-4B 模型大小只有约 2.5 GB，加载速度和推理延迟都在可接受范围内。

实际开发中最大的挑战不是模型推理本身，而是**依赖版本管理**。多个模型（VLM、TTS、ASR）对 `transformers` 等基础库的版本要求不完全一致，需要仔细处理兼容性问题。独立子进程的方案虽然不够优雅，但确实有效。

### 后续优化方向

- 补充更多真实场景的测试素材，进一步调优 VLM prompt 的稳定性
- 探索在 Intel GPU/NPU 上的加速效果
- 增加更多场景模板（校园快递柜、物业报修、酒店客房服务等）
- 在 Copaw 创空间中完整跑通 Skill 并补充运行截图
- 尝试接入视频流，支持实时交付状态监控
